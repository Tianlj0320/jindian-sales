## 一、项目概述

### 1.1 项目背景

金典软装（杭州古墩路欧亚达家居广场）主营窗帘、墙纸墙布、硬包、岩板定制业务，月均订单约 100 单，现有 3-5 名员工。随着业务增长，纯手工记账和 Excel 管理模式已无法满足需求，现有 C/S 架构系统也无法支持移动办公。急需一套轻量级 B/S 架构销售管理系统。

核心痛点：

| 问题 | 说明 |
|------|------|
| **手工台账效率低** | 订单、库存、财务全靠手记和 Excel，易出错难追溯 |
| **C/S 架构限制** | 必须在门店内网使用，无法外出办公、移动查单 |
| **无手机端** | 安装工无法手机接单、客户无法扫码查进度 |
| **ERP 过于笨重** | 市面通用 ERP 不适合月均 100 单的规模 |
| **无 AI 能力** | 客户选窗帘时看不到实际挂装效果，全靠想象 |

### 1.2 项目目标

打造一套覆盖窗帘零售全流程的新一代 B/S 架构管理系统，实现：

| 目标 | 说明 |
|------|------|
| **全面上云** | 数据库+应用服务器部署在云端，PC/手机/平板随处可用 |
| **全流程数字化** | 销售→采购→加工→安装→售后全链路在线管理 |
| **移动化** | 销售导购、安装工、客户、店长四端覆盖 |
| **AI 赋能** | 虚拟展示、AI 效果图、智能推荐 |
| **多门店支持** | 一套系统管理多个门店，数据隔离+集团汇总 |

### 1.3 用户角色

| 角色 | 使用端 | 核心职责 |
|------|--------|----------|
| 老板/管理员 | PC 后台 + 手机 H5 | 全局数据查看、财务审批、系统配置 |
| 导购 | PC 后台 + 手机 H5 | 客户管理、开单收款、订单跟踪 |
| 采购员 | PC 后台 | 采购下单、入库确认 |
| 安装工 | 手机 H5（独立页面） | 接单、确认安装、备注 |
| 客户 | 手机 H5（扫码进入） | 查订单进度 |

### 2.2 技术选型（已确认）

根据需求规格说明书 V2.2 和当前开发进度，技术栈如下：

| 层次 | 技术 | 说明 |
|------|------|------|
| 后端框架 | Python FastAPI + SQLAlchemy | 轻量高效，适合月均100单规模 |
| 数据库 | SQLite（当前）→ PostgreSQL（未来迁移） | 当前单机部署免运维，数据量增长后升级 |
| 前端 PC | Vue.js 3 + Element Plus + Vite | 有构建的现代开发模式，组件化开发 |

| 认证 | JWT（72小时有效） + bcrypt | 简单可靠 |
| 部署 | 阿里云 ECS + uvicorn | 单机部署，低运维成本 |

```
  "message": "success",
  "data": { ... }     // 业务数据

分页格式:
{
  "code": 0,
  "message": "success",
  "data": {
    "items": [...],
    "total": 100,
    "page": 1,
    "page_size": 20

```

### 3.3 销售管理（核心模块）

#### 3.3.1 订单全流程

**V2.2 标准的 8 态流转：**

```
created(待确认) → confirmed(已确认) → producing(生产中) → shipped(已发货)
                                                          ↓
completed(已完成) ← installing(安装中) ← ← ← ← ← ← ← ← ←
    ↓
    ├→ cancelled(已取消)  — 从 confirmed/producing 可转到
    └→ returned(已退回)   — 从 shipped/installing 可转到
```

**当前系统已扩展为 12 态状态机（V3.0 增强）：**

```
initial(待量尺) → measured(已量尺) → confirmed(已确认) → split(已拆分)
    → purchasing(待采购) → partial_in(部分到货) → stocked(已入库)
    → processing(生产中) → completed(已完工)
    → install_scheduled(已派工) → installed(安装中) → accepted(已完成)
    
    ├→ after_sale(售后中)  — 终端状态
    └→ cancelled(已取消)   — 终端状态
```

> 12 态完全覆盖 8 态的能力，并增加了量尺管理、采购拆分、部分到货、派工调度等细分环节。根据实际使用习惯，可随时简化回 8 态。

#### 3.3.2 订单关键数据

**订单头：**
- 订单号（自动生成：YYMMDD+4位序号）
- 客户信息

- 金额：标价 → 优惠 → 抹零 → 实付
- 收款：累计已收 / 欠款
- 交期 / 送货地址 / 安装时间

#### 3.3.3 收款管理

支持多期收款：

| 期次 | 比例 | 说明 |
|------|------|------|
| 定金 | 30%～50% | 下单时收取 |
| 中期款 | 0%～40% | 到货/安装前收取 |
| 尾款 | 10%～30% | 安装验收后收取 |

每笔收款记录：收款金额、收款方式（现金/微信/支付宝/刷卡/转账）、收款日期、收款人。

### 3.4 采购/进货管理

#### 3.4.1 采购单生成

**自动拆分**：订单确认后，按明细中的供应商分组，自动生成采购单。
- 同一供应商的物料合并为一张采购单
- 无供应商的物料归入"待定"组

**手动创建**：支持手动录入采购单（库存补货、配件采购等）

#### 3.4.2 采购流程

```
待采购 → 已下单 → 部分到货 → 全部到货
```

- 预计到货日期管理
- 到货入库：支持部分入库，自动更新库存
- 采购单关联订单，便于追溯

#### 3.4.3 供应商对账

- 按时间/供应商查询采购汇总
- 应付账款统计
- 采购价格变动分析

### 3.5 库存/仓库管理

#### 3.5.1 仓库管理

- 多仓库设置（总店仓库、分店仓库）
- 仓库间调拨（调拨单 + 流水记录）
- 库存初始化

#### 3.5.2 库存操作

| 操作 | 说明 |
|------|------|
| 入库 | 采购到货入库、退货入库、盘点入库 |
| 出库 | 生产领料、直接销售出库、盘点出库 |
| 调拨 | 仓间调拨出库/入库 |
| 盘点 | 盘点计划 → 盘点录入 → 差异调整 |

#### 3.5.3 库存预警

- 低于安全库存：红色预警
- 高于最高库存：黄色提示
- 滞销产品：超过N天无出库记录

#### 3.6.2 生产任务

- 订单到达"已入库"状态后自动生成生产任务

- 生产完成后进入"已完工"状态

#### 3.6.3 生产反馈

| 反馈类型 | 说明 |
|----------|------|

### 3.7 安装管理

#### 3.7.1 安装资源管理

- **安装队**：团队名称、负责人、联系电话、成员数

### 7.2 功能一：窗帘虚拟展示

**场景**：客户在店内选择面料后，通过摄像头拍摄窗户或上传窗户照片，AI 将窗帘布料"挂"到窗户上实时预览。

**技术方案**：
1. 前端拍摄/上传窗户照片
2. 调用 AI 服务 `/api/v1/ai/virtual-display`
3. Stable Diffusion Inpainting 或 Segment Anything 识别窗户区域
4. 将布料纹理贴合到窗户区域（考虑褶皱效果）
5. 返回合成后的效果图

**用户流程**：
```
选择面料 → 拍摄窗户 → AI 生成效果 → 客户确认 → 加入订单
```

### 7.3 功能二：AI 效果图生成

**场景**：客户想看到不同风格、不同颜色的窗帘在自己房间的整体效果。

**技术方案**：
1. 客户上传房间全景照片
2. 选择风格（现代/北欧/中式/欧式/日式）
3. 选择窗帘类型（布帘/纱帘/卷帘）
4. 调用 Midjourney API 或 Stable Diffusion
5. 生成 3-4 张不同风格的效果图

### 7.4 功能三：智能推荐

- 基于客户浏览记录和购买历史推荐搭配方案
- 窗帘 + 纱帘 + 轨道的套餐推荐
- 颜色搭配建议（墙面颜色 → 窗帘颜色推荐）
- 畅销产品推荐

### 7.5 AI 服务接口

```
POST   /api/v1/ai/virtual-display     # 虚拟展示(窗户照片+布料)
       Request: { image: file, fabric_id: int, window_rect?: {x,y,w,h} }
       Response: { result_url: string }

POST   /api/v1/ai/room-design          # 房间效果图
       Request: { image: file, style: string, curtain_type: string }
       Response: { results: [url1, url2, url3] }

POST   /api/v1/ai/recommend            # 智能推荐
       Request: { room_type: string, wall_color: string, style: string }
       Response: { products: [...], combinations: [...] }

POST   /api/v1/ai/color-match          # 配色建议
       Request: { base_color: string, style: string }
       Response: { palettes: [...] }
```

### 9.2 Docker 容器化

```yaml
version: '3.8'

services:
  mysql:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: ${DB_PASSWORD}
      MYSQL_DATABASE: curtain_erp
    volumes:
      - mysql_data:/var/lib/mysql
    ports:
      - "3306:3306"
    restart: always

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    restart: always

  backend:
    build: ./backend
    ports:
      - "8080:8080"
    environment:
      SPRING_DATASOURCE_URL: jdbc:mysql://mysql:3306/curtain_erp
      SPRING_REDIS_HOST: redis
    depends_on:
      - mysql
      - redis
    restart: always

  ai-service:
    build: ./ai-service
    ports:
      - "8000:8000"
    environment:
      OSS_ENDPOINT: ${OSS_ENDPOINT}
      OSS_ACCESS_KEY: ${OSS_ACCESS_KEY}
    restart: always

  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/conf.d:/etc/nginx/conf.d
      - ./frontend/dist:/usr/share/nginx/html
    depends_on:
      - backend
      - ai-service
    restart: always

volumes:
  mysql_data:
```

```
├── 采购管理
```
