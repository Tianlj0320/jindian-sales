## 功能需求

### 1. 订单模块（orders）

#### 1.1 订单状态推进 API

**接口**：`POST /api/orders/{order_id}/advance`

**功能**：自动识别当前 `status_key`，推进一步到 `STATUS_STEPS` 的下一位。

**状态流转规则**：
```
created → confirmed → split → purchasing → stocked → processing
→ production_exception → completed → install_order_generated
→ shipped → installed → accepted
(+ cancelled)
```

**特殊处理**：
- `completed → install_order_generated`：自动生成安装单（INS 单号）
- `installed → accepted`：同步更新对应 `InstallationOrder` 状态为「已验收」
- 终态保护：`accepted` / `cancelled` 无法推进

**响应**：包含推进后的新状态和操作结果。

#### 1.2 安装单自动生成

**触发条件**：订单状态推进到 `install_order_generated` 时自动创建 `InstallationOrder`

**带入字段**：
- `receivable_amount` = `Order.amount`
- `received_amount` = `Order.received`
- `unpaid_amount` = `Order.debt`

#### 1.3 订单编辑字段补全

前端 `saveEditOrder` items payload 补全以下 8 个字段：
- `supplier_id` — 供应商 ID
- `material_type` — 材质类型
- `fold_ratio` — 褶皱倍数
- `open_type` — 开合方式
- `style_code` — 款式代码
- `process_desc` — 加工描述
- `is_custom` — 是否定制
- `note` — 明细备注

同时补全 `install_time_slot`（安装时间段）。

### 2. 采购模块（purchase_orders）

#### 2.1 采购单到货自动入库

**触发**：`PATCH /api/purchase-orders/{po_id}` 状态推进到「全部到货」

**逻辑**：
1. 读取 `PurchaseOrder.items`（JSON 格式）
2. 遍历每个 item → 写入 `WarehouseRecord`（`record_type=in`）
3. 执行 `UPDATE products SET stock = stock + qty WHERE id = ?`
4. 回写 `PurchaseOrder.arrived_date`

**响应** data 中附加：
```json
{
  "inbound": {
    "success": true,
    "inbound_count": 3,
    "details": [...]
  }
}
```

#### 2.2 拆分时回填 product_id

**逻辑**：
- 拆分订单生成采购单时，通过 `product_code` 查找 `Product` 表并回填 `product_id`
- 若 `product_code` 为空，则通过 `product_name` 查找

#### 2.3 批量拆分并发安全

- 使用 `SELECT ... FOR UPDATE` 锁避免 PO 编号并发冲突
- 查询当前最大 PO 尾号 + 排序 + 加锁

#### 2.4 拆分状态校验

只有 `status_key in ["confirmed", "split"]` 的订单才能执行拆分。

### 3. 报表模块（reports）

#### 3.1 月报状态 key 修正

**问题**：之前只排除 `cancelled/created`，导致中间态被重复计入或遗漏。

**修复**：`sales_report` / `employee_performance` / `sales_trend` / `product_rank` 四个报表改为显式列举应计入的状态：

```python
valid_statuses = [
    "confirmed", "split", "purchasing", "stocked",
    "processing", "production_exception", "completed",
    "install_order_generated", "shipped", "installed", "accepted"
]
query = query.filter(Order.status_key.in_(valid_statuses))
```

### 4. 财务模块（finance）

#### 4.1 付款记录关联采购单

- `FinanceRecord` 增加 `purchase_order_id` 字段
- `pay` 接口支持传入 `purchase_order_id`
- 付款成功后更新 `PurchaseOrder.paid_amount += amount`

#### 4.2 收款强制 order_id 追溯

- `receive` 接口拒绝既无 `order_id` 又无 `order_no` 的请求
- 若有 `order_no` 但无 `order_id`，自动查找并回填

---

### 5. 安装模块（installer）

#### 5.1 两套安装记录统一

**设计明确**：
- `InstallationOrder` = 管理台主数据源
- `InstallTask` = 安装工 App 用

**逻辑**：
- 安装工完成任务（`completeTask`）时，优先更新 `InstallationOrder.status = "已验收"`
- 不再直接写 `Order.status_key`，改为通过 `InstallationOrder` 间接同步
- 兜底逻辑：有 `InstallationOrder` 时走它，无时直接更新 Order

---

### 6. 码表模块（dicts）

#### 6.1 码表前端持久化

**后端新增 3 个端点**：
- `POST /api/dicts/items` — 新增条目
- `PUT /api/dicts/items` — 更新条目
- `DELETE /api/dicts/items` — 删除条目（按 `category_key` + `item_key` 定位）

**前端策略**（`12-dict.js`）：
- `saveDictItem` → 先调后端 API，失败时降级保留本地 + 显示错误提示
- `delDictItem` → 先调后端 API，失败时降级保留本地 + 显示错误提示

---

### 7. 前端 UI 优化

#### 7.1 页面固定高度 + 垂直滚动

- 各模块列表页面外层容器设置固定高度
- 使用 `overflow-y: auto` 实现垂直滚动
- 高度建议：`calc(100vh - 导航栏高度)`

#### 7.2 条件选择框筛选修复

- 修复筛选参数传递到 API
- 确保选择筛选条件后点击搜索/查询，列表正确更新
