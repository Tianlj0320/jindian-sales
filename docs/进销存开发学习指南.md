# 进销存系统开发学习笔记

> 为金典软装窗帘销售系统 V3.0 编写的自我提升指南，基于行业最佳实践。

---

## 一、进销存系统的核心本质

进销存 = **采购（进） + 销售（销） + 库存（存） + 财务（财）**

```
采购到货 → 入库登记 → 库存增加 → 销售出库 → 库存减少 → 收款核销
                ↑                                      ↓
            供应商应付 ←──────────────────────────── 客户应收
```

**四流合一**：物流（货物）、资金流（钱）、发票流（票）、信息流（数据）

## 二、成熟的进销存系统必须具备的模块

### 2.1 主数据管理（MDM）

| 模块 | 核心内容 | 当前系统 |
|------|---------|---------|
| 商品管理 | 编码规则、分类、规格、单位、价格体系（进价/售价/最低售价）、辅助单位 | ⚠️ 有但不完整，缺价格三角 |
| 供应商管理 | 编码、联系人、结算方式、信用额度、交货周期 | ✅ 有 |
| 客户管理 | 编码、分类（经销商/散户）、信用额度、价格等级、地址管理 | ⚠️ 有但不完整 |
| 仓库管理 | 多仓库支持、库位管理、默认仓库 | ⚠️ 只有虚拟主仓 |

### 2.2 采购管理

| 功能 | 成熟做法 | 当前系统 |
|------|---------|---------|
| 采购订单 | 关联供应商，录入商品+数量+单价+预计到货日期 | ⚠️ 有但不完整 |
| 分批收货 | 支持部分到货，分次录入每次到货数量 | ✅ 已有（需完善）|
| 到货入库 | 自动增加库存，写入库存流水 | ✅ 已有 |
| 采购退货 | 走红字单据，扣减库存 | ❌ 缺失 |
| 供应商对账 | 应付账款明细，按供应商汇总 | ❌ 缺失 |

**核心概念：采购入库单是正式单据，确认收货后才入库存**

### 2.3 销售管理

| 功能 | 成熟做法 | 当前系统 |
|------|---------|---------|
| 销售订单/报价单 | 可先报价后转订单，支持价格策略 | ⚠️ 有订单，缺报价单 |
| 销售出库/发货 | 扣减库存，生成出库记录 | ⚠️ 订单推进即扣库存，缺少独立出库单 |
| 销售退货 | 红字销售单，退货入库 | ❌ 缺失 |
| 价格管理 | 客户等级价、最近价、批量价、最低售价保护 | ❌ 缺失 |
| 欠款控制 | 信用额度检查、超信用度需审批 | ❌ 缺失 |

**核心概念：库存"预占→确认→扣减→回滚"的一致性策略**

### 2.4 库存管理（最核心）

| 功能 | 成熟做法 | 当前系统 |
|------|---------|---------|
| 库存台账 | 每笔出入库形成一条账目，`期初+入库-出库=期末` | ✅ 有（inventory_flow）|
| 批次管理 | 窗帘布料需要按批次/卷号管理 | ❌ 缺失 |
| 库位管理 | 仓库→库区→货架→货位 | ❌ 缺失 |
| 库存预警 | 安全库存、最低库存、呆滞料预警 | ⚠️ 有简单版 |
| 盘点 | 定期盘点，调价单生成报损报溢 | ❌ 缺失 |
| 调拨 | 仓库之间转移，两条库存流水（一出一入） | ⚠️ 有但不完整 |
| 库存查询 | 实时库存快照，支持按仓库/商品/批次筛选 | ⚠️ 有但不完整 |

**关键：库存变动必须通过单据驱动，绝不允许直接改库存数字！**

### 2.5 财务管理

| 功能 | 成熟做法 | 当前系统 |
|------|---------|---------|
| 应收管理 | 客户欠款台账，收款核销，支持预收款 | ⚠️ 有但不完整 |
| 应付管理 | 供应商欠款台账，付款核销 | ⚠️ 有但不完整 |
| 收款核销 | 收款可指定订单核销，支持部分核销 | ⚠️ 有但不完整 |
| 流水记账 | 每笔收款/付款/费用形成日记账 | ⚠️ 有但不完整 |
| 财务报表 | 利润表、资产负债表、应收账款账龄 | ❌ 缺失 |

**资金流必须与单据绑定：先有销售单才有应收款，收款必须核销应收款！**

## 三、当前系统与成熟进销存的差距

### 高优先级差距（影响日常使用）

1. **缺少独立的销售出库/发货确认节点**
   - 当前：订单推进就自动扣库存
   - 问题：没有"通知发货→实际出库→确认送达"的流程
   - 改进：增加 `dispatched` 状态，或独立发货单

2. **缺少采购退货流程**
   - 当前只能收货，不能退货给供应商
   - 改进：新增采购退货单（红字入库）

3. **缺少销售退货流程**
   - 当前只能售出，不能处理客户退货
   - 改进：新增销售退货单（红字出库）

4. **缺少盘点功能**
   - 实际库存和系统库存可能有差异
   - 改进：增加盘点单，生成报损/报溢单

5. **客户/供应商缺少编码体系**
   - 当前用自动生成，格式不统一
   - 改进：建立编码规则，如 `KH20260506001`

### 中等优先级差距（影响效率）

6. **缺少价格策略**
   - 不同客户等级、不同采购量没有差异化价格
   - 最低售价保护缺失（成本价以下无法拦截）

7. **缺少信用额度控制**
   - 客户超信用度仍可下单
   - 改进：下单前检查欠款+本次订单金额是否超信用度

8. **缺少多仓库管理**
   - 当前只有虚拟主仓
   - 窗帘销售业务可能需要"门店仓+仓库"分离

9. **缺少批次/序列号管理**
   - 窗帘按卷卖，每卷的米数不同
   - 改进：批次表记录每卷长度，出库时按批次扣减

## 四、如何从当前状态改进

### 改进原则

1. **单据驱动库存**：所有库存变动必须通过单据（入库单/出库单/调拨单），不允许直接改 `product.stock`
2. **事务原子性**：库存变动和资金变动必须在同一事务中
3. **状态机约束**：每种单据有固定的状态流转，不允许跳状态
4. **审计追溯**：每笔库存变动记录操作人、时间、关联单据

### 实施顺序建议

```
第一阶段（紧急）：
1. 完善采购退货流程（已有收货，补退货不难）
2. 增加销售退货流程
3. 增加盘点功能

第二阶段（重要）：
4. 独立的销售发货确认（不等同于订单推进）
5. 最低售价保护
6. 客户信用额度检查

第三阶段（优化）：
7. 多仓库支持
8. 批次管理
9. 完整财务报表
```

## 五、单据设计规范（参考）

### 标准进销存单据必须包含的公共字段

| 字段 | 说明 |
|------|------|
| `单据编号` | 全局唯一，格式如 `PO20260506001`（采购）、`SO20260506001`（销售）|
| `制单人` | 创建人 |
| `制单日期` | 创建时间 |
| `审核人` | 谁审批确认 |
| `审核日期` | 审批时间 |
| `关联单据` | 关联的原始单据ID（如退货关联原采购/销售单）|
| `备注` | 自由文本 |
| `状态` | `draft（草稿）→ confirmed（已确认）→ completed（已完成）` |

### 库存流水规范

每笔库存变动写入 `inventory_flow`，必须包含：
- `qty_before` / `qty_change` / `qty_after`：变动前后库存，清晰可追溯
- `ref_type` + `ref_id`：关联单据类型和ID
- `flow_type`：IN/OUT/TRANSFER_IN/TRANSFER_OUT/ADJUST
